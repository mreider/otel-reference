Foundations
01

Telemetry

Two systems can run the same code. One fails in silence—no clues, no trail. The other fails and tells you why, because it was built to emit data about itself.

That data is telemetry: measurements and records a system produces about its own behavior. Without it, a running service is a black box. With it, you can see inside.

Telemetry is not debugging output or error logs alone. It is the complete set of signals a system emits so that operators can understand what is happening, when, and where.

02

Three Signals

OpenTelemetry organizes telemetry into three signal types. Each answers a different class of question about your system.

Metrics Measure how much, how fast, how often. Metrics aggregate events into numbers over time intervals.
Traces Follow a single request through multiple services. Traces narrate the journey and reveal where time is spent.
Logs Record specific events with messages and context. Logs capture discrete moments—what happened at a particular point in time.
03

Metrics

Imagine 300 events scrolling past per second. No human can read them all. A metric solves this by aggregating raw events into measurements taken over time intervals—a running count, a rate, a statistical summary.

Where raw events show individual moments, metrics reveal patterns: is traffic growing? Is latency spiking? Are errors increasing? The answer is always a number measured over a window of time.

04

Metric Types

Different questions require different kinds of measurement. OpenTelemetry defines three fundamental metric types.

Counter A value that only goes up. Total requests served, bytes sent, errors encountered. Monotonically increasing.
Gauge A value that goes up and down. Current temperature, active connections, memory usage. A snapshot of the present.
Histogram A distribution of values across buckets. Request latency, payload sizes. Reveals shape—are most requests fast with a few slow outliers, or is everything uniformly slow?
05

Traces

A metric dashboard tells you latency is high. Logs show individual errors. But neither answers: which service in the chain is the bottleneck?

A trace follows one request from entry to exit across every service it touches. It records the path the request took and the time spent at each step. When a checkout request passes through gateway → api → orders → db, the trace shows you that the orders service took 700ms of the total 1.2s.

gateway |----- 1200ms ------| api |--- 950ms ---| orders |-- 700ms --| db |- 80ms -|
Anatomy of Telemetry
06

Spans

A trace is not a single unit—it is composed of spans. Each span represents one operation: an HTTP handler, a database query, an RPC call. Spans record a start time, a duration, and a status (ok or error).

Spans nest inside each other, forming a tree. A parent span might represent an API request; its children might represent the database call and cache lookup that happen within it. By examining the tree, you can see exactly where time is spent and which operation failed.

07

Logs

Traces tell you where a failure occurred. Metrics tell you how often. Logs tell you why.

A log record captures a discrete event at a specific moment: an error message, a stack trace, the exact input that caused a failure. Each record carries a timestamp, a severity level, and a body. When connected to a span, a log transforms from an isolated line of text into a moment within a larger story.

08

Attributes

A span that says "HTTP request, 200ms" is not very useful. But a span that says "GET /api/orders, 200ms, status=500, user_id=42, region=us-east" tells you everything you need.

Attributes are key-value pairs attached to any signal—spans, metrics, or logs. They provide the context that makes telemetry actionable: the HTTP method, the endpoint, the response status, the customer tier. In metrics, attributes are sometimes called dimensions because they let you slice and filter aggregated data.

09

Resources

Attributes describe what happened. Resources describe who reported it.

A resource identifies the entity producing telemetry: the service name, deployment environment, host, version, and container ID. When ten services emit the same metric, it is the resource that tells you which service each data point came from.

service.name = checkout-svc service.version = 2.4.1 deployment.env = production host.name = prod-checkout-7a3b container.id = a1b2c3d4e5f6
10

Semantic Conventions

If one team names an attribute http.status and another calls it response_code, a single query cannot find both. Independent naming leads to fragmented data.

Semantic conventions are OpenTelemetry's shared naming rules. They define standard attribute names for common concepts: http.request.method, http.response.status_code, db.system, rpc.service. When every service follows the same conventions, one query works across your entire fleet.

Infrastructure
11

Instrumentation

Code does not emit telemetry by itself. Instrumentation is the act of adding telemetry-producing code to a program, bridging the gap between execution and observability.

OpenTelemetry provides two approaches. Automatic instrumentation hooks into frameworks and libraries to emit spans and metrics without code changes. Manual instrumentation uses the OTel API directly—creating spans, recording metrics, emitting logs—for custom business logic that libraries cannot cover.

12

Exporters

Instrumentation creates telemetry inside a process. To leave the process, telemetry must be serialized into a structured, standardized format and sent somewhere.

An exporter does this: it takes the in-memory spans, metrics, and logs and transmits them over a protocol to a backend or collector. OTLP (OpenTelemetry Protocol) is the standard wire format, ensuring any compatible receiver can understand the data regardless of the language that produced it.

13

Data Formats

The same telemetry data can be encoded in different ways. The format is the envelope, not the message.

OTLP/gRPC Binary protobuf over gRPC. Efficient, bidirectional, the default for production.
OTLP/HTTP Protobuf or JSON over HTTP. Easier to route through proxies and load balancers.
JSON Human-readable. Useful for debugging and development, but larger on the wire.

Different backends may require different formats. Exporters handle the translation.

14

The Collector

Without a central hub, every service must know about every backend—N services × M backends = N×M connections. The OpenTelemetry Collector reduces this to N + M: services send to the collector, and the collector sends to backends.

The Collector has three stages:

Receivers Accept incoming telemetry in various formats (OTLP, Jaeger, Prometheus, etc.).
Processors Transform data in flight—batch, filter, enrich, sample, drop.
Exporters Send processed data to one or more backends.
15

The Pipeline

A pipeline is the Collector's blueprint, defined in YAML. It wires together receivers, processors, and exporters into a data flow for each signal type.

service: pipelines: traces: receivers: [otlp] processors: [batch, tail_sampling] exporters: [otlp/jaeger, debug] metrics: receivers: [otlp, prometheus] processors: [batch] exporters: [otlp/backend]

Each signal type (traces, metrics, logs) gets its own pipeline with its own processing rules. You can run multiple pipelines simultaneously, each with different receivers, processors, and destinations.

Scale & Topology
16

Sampling

At low traffic, you can keep every trace. At scale, you cannot—storage and processing costs grow linearly with volume. Sampling is the practice of deciding which traces to keep and which to drop.

Head sampling Decide at the start of a trace (e.g., keep 10% randomly). Simple but blind to outcomes.
Tail sampling Decide after a trace completes. Keep all errors and slow requests, drop routine successes. Smarter but requires buffering.

The goal is to preserve the traces that matter—errors, high latency, unusual patterns—while reducing the volume of routine, healthy traffic.

17

Service Maps

No one draws the architecture diagram—the traces draw it. Each span records which service made a call and which service received it. By analyzing trace data, you can automatically reconstruct the topology of your system: which services exist, how they connect, and where the dependencies are.

Service maps are not manually maintained documentation. They are a live projection of actual traffic patterns, always current, always accurate.

18

Correlation

Each signal tells a different part of the story. Metrics say that something is wrong. Traces show where in the request path. Logs explain why.

The trace_id is the thread that connects all three. A metric can include an exemplar pointing to a trace ID. A log record can carry the trace ID and span ID of the operation that produced it. With this shared identifier, you can navigate from an alert to the exact request to the exact error message—across all three signals.

19

The Full Picture

Real incident investigation uses all three signals together. A typical workflow:

Metrics trigger an alert: error rate spiked at 14:32.

Traces narrow the scope: the payment service is timing out on requests to the database.

Logs reveal the cause: "connection pool exhausted—max connections reached."

No single signal is sufficient. Metrics establish when and what. Traces pinpoint where. Logs explain why. Together, they reveal root cause.

Advanced Concepts
20

Events

Not every moment needs its own span. Events are lightweight, timestamped annotations attached to a span. They mark significant moments—a cache miss, a retry attempt, a validation failure—without the overhead of creating child spans.

Each event has a name, a timestamp, and optional attributes. They are the right tool when you need to record that something happened within a span, but do not need to measure its duration.

21

Trace Context

When a request crosses a service boundary, how does the next service know it is part of the same trace? Through context propagation.

The W3C traceparent header carries four fields in a single HTTP header:

traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01 ││ ││ ││ ││ version trace-id span-id trace-flags

Every service extracts this header, continues the trace, and propagates it to the next call. This is the invisible thread that holds a distributed trace together.

22

Baggage

Trace context carries telemetry identity (trace ID, span ID). Baggage carries application data across services using the same propagation mechanism.

A frontend might set user.tier=premium in baggage. Every downstream service receives it automatically—no extra database lookups, no additional API calls. Baggage values can be read by span processors and added as attributes, enabling business-context-aware telemetry throughout the request.

Baggage travels with every request. Keep values small and be aware that all downstream services can read them.

23

Span Links

Not all relationships are parent-child. A message queue consumer processes a message that was produced by a different trace. A batch job aggregates work from many requests. These are related but not nested.

Span links connect spans across traces without changing ownership. They say "this span is related to that span" while preserving each trace's independent lineage. Common uses: async message processing, fan-in aggregation, and scheduled jobs triggered by earlier requests.

24

Log Correlation

An OTel log appender automatically injects trace context—trace_id and span_id—into every log record emitted within an active span. This transforms logs from isolated text into navigable entries.

From any log line, you can jump to the full distributed trace. From any span, you can see every log that was emitted during its execution. The three signals stop being separate databases and become one connected story.

25

Exemplars

Metrics aggregate. That is their strength—and their limitation. When a histogram shows a latency spike, which specific request caused it?

Exemplars are links from aggregated metric data points back to the individual traces that produced them. A histogram bucket recording 1.2s latency can carry an exemplar pointing to the exact trace ID of one of those slow requests. The metric tells you something is slow; the exemplar takes you to this specific slow request.

26

Resource Detectors

Resource detectors automatically discover environment attributes at startup: the host name, cloud provider, Kubernetes namespace, OS type, container ID. No hardcoding, no config files.

The SDK runs registered detectors at initialization and merges their findings into the resource. The only attribute teams typically set manually is service.name—everything else can be auto-detected from the runtime environment.

Host detector host.name, host.arch
OS detector os.type, os.description
Cloud detector cloud.provider, cloud.region, cloud.availability_zone
K8s detector k8s.namespace.name, k8s.pod.name, k8s.deployment.name
27

Collector Transforms

The Collector's transform processor reshapes telemetry in flight through configuration alone—no application redeployment required.

Common operations:

Delete attributes Strip PII (email addresses, user names) before data leaves your infrastructure.
Add attributes Enrich all telemetry with environment tags (region, cluster, team).
Filter spans Drop noisy health-check spans that add volume without insight.
Rename Normalize attribute names to match semantic conventions without changing application code.
28

Profiling

Traces show where time is spent. Logs show what happened. Profiling shows why at the code level—by capturing the actual stack frames executing on the CPU.

OpenTelemetry's profiling signal (currently experimental) links profiles to traces via span context. A slow span does not just tell you "this operation took 700ms"—it connects to a flamegraph showing exactly which functions consumed that time. One click from a slow span to the code that caused it.

Profiling is an emerging signal in OpenTelemetry. The specification and SDK support are still under active development.
